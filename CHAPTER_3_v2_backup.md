# ГЛАВА 3. РЕАЛИЗАЦИЯ И ТЕСТИРОВАНИЕ ПРОГРАММНОГО КОМПЛЕКСА AWS OPTIMIZER

## 3.1 Архитектура системы и выбор технологического стека

### 3.1.1 Структура многоуровневой архитектуры

Разработанный программный комплекс AWS Optimizer построен на основе классической трёхуровневой архитектуры (трёхслойной модели), обеспечивающей чёткое разделение ответственности и возможность независимого масштабирования каждого уровня.

Система состоит из следующих компонентов:

**Уровень представления (клиентская часть).** Веб-приложение, разработанное на базе фреймворка React.js версии 19 с использованием языка TypeScript. Приложение размещается на локальном хосте порт 5173 при разработке и в рабочей среде через CDN (CloudFlare). Клиентская часть обеспечивает визуализацию результатов сканирования инфраструктуры, представление метрик в виде интерактивных графиков и табличных форм, а также реализует взаимодействие пользователя с системой.

**Уровень бизнес-логики (серверная часть).** REST API-сервис, реализованный на базе платформы Node.js с использованием фреймворка Express.js и языка TypeScript. Сервис развёртывается на локальном хосте порт 5000 при разработке. На данном уровне реализованы следующие функции: аутентификация и авторизация пользователей, инициирование и координирование процессов сканирования инфраструктуры облачного провайдера, применение встроенных правил анализа безопасности и финансовой эффективности, управление сессиями пользователя, взаимодействие с внешними сервисами (AWS API, LLM-модель Groq).

**Уровень хранения данных (база данных).** Реляционно-распределённая база данных MongoDB, функционирующая на локальном хосте порт 27017 при разработке и на управляемом сервисе MongoDB Atlas в рабочей среде. База данных хранит учётные данные пользователей, историю выполняемых сканирований инфраструктуры, обнаруженные уязвимости и рекомендации, метрики безопасности и финансовой эффективности, а также историю диалогов с интеллектуальным помощником.

Взаимодействие между уровнями реализуется через стандартные протоколы передачи данных:
- Между клиентской частью и серверной частью: HTTP/HTTPS, формат данных JSON
- Между серверной частью и базой данных: Драйвер MongoDB (TCP/IP соединение)
- Между серверной частью и внешними API: HTTPS, JSON (для AWS SDK и Groq API)

## 3.2 Программная реализация основных модулей

### 3.2.1 Модуль аутентификации и авторизации

Система безопасности доступа построена на основе токен-ориентированного подхода с использованием JWT (JSON Web Tokens).

**Процесс регистрации пользователя:**

При регистрации пользователь предоставляет адрес электронной почты и пароль. Введённый пароль подвергается хешированию с использованием алгоритма bcrypt с 10 раундами итераций (что соответствует времени обработки приблизительно 100 миллисекунд). В базу данных сохраняется хеш пароля, оригинальный пароль не сохраняется. Использование солей (автоматически генерируемые bcrypt) предотвращает атаки на основе предварительно вычисленных хеш-таблиц (rainbow tables).

**Процесс аутентификации (вход):**

При входе в систему пользователь повторно вводит адрес электронной почты и пароль. Пароль хешируется тем же алгоритмом и сравнивается с сохранённым хешем. Если совпадение подтверждено, генерируется JWT-токен, подписанный алгоритмом HS256 (HMAC-SHA256) с секретным ключом сервера.

**Параметры JWT-токена:**

- Время жизни токена: 24 часа
- Алгоритм подписи: HS256
- Содержимое payload: userId, email, iat (время создания), exp (время истечения)
- Хранение на клиенте: localStorage веб-браузера

**Авторизация при обращении к API:**

Все защищённые эндпоинты API требуют предоставления действительного JWT-токена в заголовке HTTP Authorization (формат: "Bearer <token>"). Серверная часть проверяет подпись токена, срок действия и наличие требуемых claims перед обработкой запроса.

### 3.2.2 Модуль сканирования инфраструктуры AWS

Модуль сканирования реализует автоматизированный процесс обнаружения и классификации ресурсов облачной инфраструктуры, а также выявления потенциальных уязвимостей и неэффективно используемых ресурсов.

**Процесс инициирования сканирования:**

Пользователь предоставляет учётные данные AWS (Access Key ID, Secret Access Key) и выбирает географический регион. Учётные данные передаются через HTTPS-соединение на серверную часть и сохраняются в базе данных с использованием алгоритма шифрования AES-256. При выполнении сканирования зашифрованные данные расшифровываются в памяти сервера (не на диске) для инициализации AWS SDK клиента.

**Парадигма сканирования (параллельные запросы):**

Вместо последовательного опроса каждого сервиса AWS, система инициирует параллельные асинхронные запросы ко всем поддерживаемым сервисам (EC2 DescribeInstances, EBS DescribeVolumes, IAM ListRoles, EC2 DescribeSecurityGroups, EC2 DescribeAddresses). Использование асинхронности снижает общее время сканирования.

**Поддерживаемые типы ресурсов:**

1. EC2 Instances (виртуальные машины): тип инстанса, статус, IP-адреса, подключённые тома, теги
2. EBS Volumes (блочные хранилища): размер в ГБ, статус шифрования, снимки состояния (snapshots)
3. Elastic IPs (зарезервированные статические IP): статус привязки, текущий адрес
4. Security Groups (группы правил доступа): входящие и исходящие правила, диапазоны портов и IP-адресов
5. IAM Roles and Policies (роли управления доступом): назначенные политики, разрешённые действия

**Временная характеристика сканирования:**

Время выполнения сканирования варьируется от 3 секунд (для малых инфраструктур с десятками ресурсов) до 85 секунд (для крупных инфраструктур с десятками тысяч ресурсов). Данное варьирование обусловлено объёмом обрабатываемых данных и ограничениями частоты запросов (rate limiting) со стороны AWS API.

### 3.2.3 Встроенный набор правил анализа

Система содержит встроенный набор из более чем 20 правил анализа, разделённых на две категории: правила выявления уязвимостей безопасности и правила выявления финансовой неэффективности.

**Уязвимости категории CRITICAL (критические):**

Уязвимости, требующие немедленного исправления вследствие исключительно высокого риска несанкционированного доступа или компрометации инфраструктуры:

- **SSH открыт для всех IP-адресов (0.0.0.0/0 на порту 22).** Правило обнаруживает Security Groups с входящим правилом, разрешающим трафик на порту 22 со всех IP-адресов. Потенциальная атака: внешний злоумышленник может перебирать пароли SSH (brute-force attack) или использовать известные уязвимости SSH-демона. Рекомендация: ограничить доступ SSH на конкретные IP-адреса (например, IP корпоративного офиса или VPN).

- **Root аккаунт без многофакторной аутентификации (MFA).** Правило проверяет конфигурацию IAM для выявления отсутствия MFA на корневом (root) аккаунте. Потенциальная атака: если пароль root аккаунта скомпрометирован, злоумышленник получает полный контроль над всеми AWS ресурсами. MFA требует предоставления второго фактора (например, одноразовый код из мобильного приложения), что существенно затрудняет несанкционированный доступ.

- **IAM политики с полным доступом (действие: "*").** Правило обнаруживает IAM политики, содержащие действие "*" (asterisk), что эквивалентно предоставлению всех прав пользователю или роли. Потенциальная атака: если access key пользователя с такой политикой скомпрометирован, злоумышленник может удалить инфраструктуру, создать дорогостоящие ресурсы (например, GPU-инстансы для криптомайнинга) или украсть конфиденциальные данные.

**Уязвимости категории HIGH (высокие):**

Уязвимости, требующие исправления в течение дней или недель:

- **EBS тома без шифрования.** Правило выявляет EBS Volumes с параметром Encrypted = false. Зашифрованное хранилище защищает данные от физического доступа к оборудованию или снятия снимков состояния.

- **S3 bucket с публичным доступом.** Правило обнаруживает S3 buckets с блоком-бланк (ACL или bucket policy, разрешающей чтение или запись всем пользователям интернета). Потенциальный риск: утечка конфиденциальных данных.

- **Устаревшие снимки состояния (старше 90 дней).** Правило выявляет EBS snapshots, последний раз модифицированные более 90 дней назад. Такие снимки часто являются результатом забытых резервных копий и не требуют сохранения.

**Уязвимости категории MEDIUM и LOW:**

К категории MEDIUM относятся обнаруженные EC2 инстансы с чрезвычайно низким использованием CPU (< 5% за последние 7 дней) — такие инстансы могут быть уменьшены в размере или остановлены. К категории LOW относятся использование устаревших типов инстансов (например, t2 вместо t3) или использование классических load balancers вместо modern ALB.

### 3.2.4 Модуль расчёта метрик и панели управления

Информационная панель (dashboard) предоставляет пользователю синтезированное представление состояния его облачной инфраструктуры и финансовой эффективности.

**Четыре основные ключевые показатели эффективности (KPI):**

1. **Общие ежемесячные расходы (Total Monthly Spend).** Рассчитывается путём агрегирования стоимостных данных для всех обнаруженных ресурсов на основе текущих тарифов AWS. Обновляется при каждом выполнении сканирования.

2. **Потраченные ресурсы (Wasted Resources).** Денежная оценка неэффективно используемых или избыточных ресурсов, полученная на основе применения встроенных финансовых правил (неиспользуемые EIP по 3.60 USD/месяц каждый, EBS тома без привязки, EC2 инстансы с CPU < 5%).

3. **Индекс безопасности (Health Score).** Композитный показатель (0–100 пунктов), рассчитываемый как взвешенная функция от количества выявленных уязвимостей различных категорий: CRITICAL (вес: -20 пунктов за каждую), HIGH (вес: -10 пунктов), MEDIUM (вес: -5 пунктов), LOW (вес: -1 пункт). Максимальный балл (100) присваивается инфраструктуре без уязвимостей.

4. **Общее количество ресурсов (Total Resources).** Числовой показатель, отражающий совокупное количество обнаруженных ресурсов AWS всех типов.

**Интерактивные графики:**

Панель управления содержит четыре интерактивных графика, построенные с использованием библиотеки Chart.js:

- Graph 1: Линейный график динамики затрат (Cost Trend) за последние 7 дней
- Graph 2: Круговая диаграмма распределения ресурсов по типам (Resource Distribution)
- Graph 3: Линейный график динамики индекса безопасности (Health Score Trend) за 7 дней
- Graph 4: Столбчатая диаграмма потраченных ресурсов по категориям

**Таблица последних сканирований:**

Таблица отображает пять последних выполненных сканирований с колонками: timestamp (дата и время сканирования), количество выявленных уязвимостей, индекс безопасности, общие расходы и статус (успешно/ошибка).

### 3.2.5 Модуль интеллектуального помощника (AI Advisor)

Модуль интеллектуального помощника реализует взаимодействие пользователя с языковой моделью на основе API Groq, предоставляя техническое руководство по исправлению выявленных уязвимостей и оптимизации расходов.

**Архитектура взаимодействия с LLM:**

Процесс формирования ответа состоит из следующих этапов:

1. **Формирование контекста (context assembly).** Серверная часть извлекает из базы данных последние пять обнаруженных уязвимостей для конкретного пользователя, а также текущие метрики финансовой эффективности. Данные структурируются в текстовое описание (промпт).

2. **Составление системной инструкции (system prompt).** Формируется инструкция для языковой модели с указанием роли ("вы являетесь экспертом в области облачной безопасности и оптимизации AWS расходов") и требованиями к качеству ответа (краткость, ориентация на действия, избегание общих фраз).

3. **Отправка запроса через API.** Сформированный пакет отправляется через REST API-интерфейс сервиса Groq. Параметры запроса установлены следующим образом:
   - Модель: mixtral-8x7b (смешанная модель из 8 экспертных подмоделей по 7 миллиардов параметров каждая)
   - temperature: 0.3 (низкое значение для детерминированных технических ответов, минимизирует риск "галлюцинаций" модели)
   - max_tokens: 512 (ограничение длины ответа)

4. **Обработка ответа.** Полученный от языковой модели ответ парсится, сохраняется в коллекцию ChatHistory базы данных MongoDB для сохранения истории диалога, и передаётся на клиентскую часть для визуализации.

**Временная характеристика:**

Время отклика системы на данном уровне варьируется от 0.5 до 1 секунды, что соответствует требованиям интерактивности веб-приложения.

**Управление историей диалога:**

Для экономии токенов LLM-модели и управления памятью серверной части система сохраняет максимум 12 последних сообщений диалога для каждого пользователя. Более старые сообщения автоматически удаляются.

### 3.2.6 Схема базы данных

База данных MongoDB содержит четыре основные коллекции:

**Коллекция User:**

```
{
  _id: ObjectId,
  email: String,
  passwordHash: String (bcrypt),
  createdAt: Date,
  lastLogin: Date,
  awsCredentialsEncrypted: String (AES-256),
  preferredRegion: String
}
```

**Коллекция Audit:**

```
{
  _id: ObjectId,
  scanId: String (unique),
  userId: ObjectId (reference to User),
  timestamp: Date,
  totalResources: Number,
  resourceBreakdown: {
    ec2: Number,
    ebs: Number,
    eip: Number,
    sg: Number
  },
  alerts: [
    {
      resourceId: String,
      severity: String (CRITICAL/HIGH/MEDIUM/LOW),
      ruleId: String,
      description: String,
      recommendation: String
    }
  ],
  metrics: {
    totalSpend: Number,
    wastedAmount: Number,
    healthScore: Number
  }
}
```

**Коллекция SecurityMetrics (ежедневные снимки состояния):**

```
{
  _id: ObjectId,
  userId: ObjectId (reference to User),
  date: Date,
  totalSpend: Number,
  wastedAmount: Number,
  healthScore: Number,
  alertCount: Number
}
```

**Коллекция ChatHistory:**

```
{
  _id: ObjectId,
  userId: ObjectId (reference to User),
  timestamp: Date,
  userMessage: String,
  aiResponse: String,
  tokensUsed: Number,
```

**Индексы для оптимизации производительности:**

## 3.3 Развёртывание системы и обеспечение отказоустойчивости

### 3.3.1 Локальная разработка с использованием Docker Compose

При разработке и тестировании приложение развёртывается локально с использованием Docker и Docker Compose, что обеспечивает:

- **Воспроизводимость среды:** Разработчики независимо от операционной системы (Windows, macOS, Linux) получают идентичную среду разработки.
- **Отсутствие зависимостей хоста:** Все сервисы (Node.js, MongoDB) запускаются в контейнерах и не требуют установки на хост-машину.
- **Быстрый старт:** Команда `docker-compose up -d` запускает полную систему (frontend, backend, database) в течение нескольких секунд.

**Состав контейнеризированной системы:**

- Backend: Node.js 20 Alpine Linux (минимальный образ размером ~150 МБ)
- Frontend: Multi-stage сборка (builder stage для компиляции TypeScript → nginx для сервирования статики)
- Database: MongoDB 6.0 официальный образ

**Конфигурация сетевого взаимодействия:**

Контейнеры подключены к внутренней Docker-сети (bridge network), что позволяет безопасному взаимодействию между контейнерами по внутренним имён хостов (service discovery).

### 3.3.2 Локальное тестирование с эмуляцией AWS (LocalStack)

Для обеспечения безопасного и экономически целесообразного тестирования разработана интеграция с проектом LocalStack, который предоставляет локальную эмуляцию AWS-сервисов.

**Поддерживаемые эмулируемые сервисы:**

- EC2 (виртуальные машины): возможность создания, запуска, остановки инстансов
- EBS (блочные хранилища): создание томов, снимков состояния
- Elastic IPs: резервирование и привязка статических адресов
- Security Groups: создание правил доступа
- IAM: управление ролями и политиками доступа

**Преимущества использования LocalStack в процессе разработки:**

| Аспект | Без LocalStack (реальный AWS) | С LocalStack |
|---|---|---|
| **Стоимость** | $50–200/месяц при интенсивном тестировании | $0 (локальное ПО) |
| **Зависимость от интернета** | Требуется постоянное соединение | Не требуется |
| **Скорость тестирования** | 3–10 сек на запрос (сетевая задержка) | < 100 мс (локальный вызов) |
| **Безопасность** | Риск утечки реальных данных/credentials | Изолированная среда |
| **Автоматизация** | Сложное восстановление после ошибок | Легко удалить и пересоздать инфраструктуру |

**Интеграция в приложение:**

Клиентская часть содержит переключатель режима "Use LocalStack" в меню Settings. При активации:
- Endpoint API переходит на http://localhost:4566 (LocalStack)
- Access Key ID и Secret Access Key заполняются тестовыми значениями ("test")
- Все API запросы маршрутизируются на локальный эмулятор вместо облака AWS

### 3.3.3 Production-развёртывание в облаке AWS с использованием Kubernetes

Архитектура production-среды построена на основе контейнерной оркестрации с использованием Amazon EKS (Elastic Kubernetes Service).

**Компоненты production-архитектуры:**

- **Kubernetes кластер (EKS):** Управляемый сервис AWS, обеспечивающий автоматическое масштабирование и восстановление после сбоев
- **Узлы кластера:** 3–5 узлов типа t3.medium (2 vCPU, 4 ГБ памяти каждый)
- **Container Registry:** Amazon ECR (Elastic Container Registry) для хранения Docker-образов backend и frontend
- **База данных:** MongoDB Atlas (управляемый сервис MongoDB) с автоматическим резервным копированием и репликацией
- **Кеширование:** Redis ElastiCache для кеширования часто запрашиваемых данных (профили пользователей, метрики панели управления)
- **Сеть и доступ:** Application Load Balancer (ALB) для распределения входящего трафика, CloudFront CDN для кеширования статических ресурсов (CSS, JavaScript, изображения)
- **Управление секретами:** AWS Secrets Manager для безопасного хранения учётных данных (JWT secret, AWS API keys, LLM API keys)

**Процесс CI/CD и развёртывания:**

1. Разработчик публикует код в репозиторий GitHub
2. GitHub Actions автоматически инициирует pipeline:
   - Запуск unit-тестов
   - Запуск integration-тестов на LocalStack
   - Запуск E2E-тестов
   - При успехе всех тестов: сборка Docker-образов
3. Docker-образы загружаются в Amazon ECR
4. Kubernetes deployment автоматически обновляется (rolling update)
5. Новые pods проходят health checks перед включением в load balancer
6. Трафик постепенно переводится на новые pods

**Rolling Update механизм:**

Процесс обновления приложения реализуется таким образом, чтобы минимизировать простой:

- На каждом этапе обновляется один pod из группы
- Новый pod запускается и проходит health checks
- Трафик переводится на новый pod
- Старый pod остаётся доступным для откатов
- Процесс повторяется для каждого pod

Общее время развёртывания: ~5 минут при нулевом простое (zero downtime deployment).

**Механизм автоматического отката (Automatic Rollback):**

Система мониторинга отслеживает ключевые метрики во время развёртывания:

- Если процент ошибок (error rate) превышает 5% в течение 2 минут → автоматический откат на предыдущую версию
- Если среднее время отклика API (p95) превышает 10 секунд → откат
- Если процент crashing pods превышает 50% → откат

Время отката: ~2 минуты.

### 3.3.4 Мониторинг, алерты и логирование

**Система сбора метрик (Prometheus):**

На backend-сервисе развёрнут Prometheus exporter, который собирает следующие метрики:

*Метрики API:*
- http_request_duration_seconds (время ответа с квантилями p50, p95, p99)
- http_requests_total (общее количество запросов по статусу кода)
- http_request_size_bytes (размер входящего запроса)
- http_response_size_bytes (размер исходящего ответа)

*Метрики базы данных:*
- mongodb_operation_duration_seconds (время выполнения операций)
- mongodb_connection_pool_size (размер пула соединений)
- mongodb_slow_queries_total (количество медленных запросов)

*Метрики приложения:*
- aws_scan_duration_seconds (время выполнения сканирования)
- groq_api_response_time_seconds (время отклика LLM)
- jwt_validation_failures_total (количество отказов при валидации токенов)

**Визуализация метрик (Grafana):**

На основе собранных Prometheus-метрик создано несколько дашбордов Grafana:

1. **System Health Dashboard:** CPU, Memory, Disk utilization каждого узла Kubernetes, pod restart count
2. **API Performance Dashboard:** Request rate, response time (p50/p95/p99), error rate, throughput
3. **Database Performance Dashboard:** Query latency, connection pool status, slow query analysis
4. **Business Metrics Dashboard:** Number of scans per day, AI queries per day, new user registrations
5. **AWS Integration Dashboard:** AWS API latency, rate limiting incidents, credential validation failures

**Система алертов:**

На основе метрик настроены автоматические алерты, отправляемые в Slack и PagerDuty:

### 3.4.5 Тестирование безопасности

**Проверка защиты от SQL Injection:**

Несмотря на использование MongoDB (нет SQL), проведены тесты на попытки injection-атак через поля ввода. Результат: все попытки блокированы, тестовые строки с спецсимволами корректно эскейпятся.

Результат: 5/5 попыток отклонено ✓

**Проверка защиты от XSS (Cross-Site Scripting):**

В поле описания уязвимости введены строки вида `<script>alert('XSS')</script>`. React автоматически экранирует содержимое, скрипт не выполняется.

Результат: 3/3 попыток экранировано ✓

**Проверка защиты от CSRF (Cross-Site Request Forgery):**

Система использует JWT-токены вместо cookie-based sessions. Попытка выполнить cross-origin POST-запрос без валидного токена отклоняется.

Результат: 2/2 попыток отклонено ✓

**Проверка шифрования AWS credentials:**

AWS credentials сохраняются в БД с использованием AES-256. Попытка прямого доступа к зашифрованным данным показывает нечитаемые байты. Расшифровка требует секретного ключа, хранящегося в AWS Secrets Manager.

**Проверка bcrypt хеширования паролей:**

Извлечённые из БД bcrypt-хеши не могут быть обратно переведены в пароли (одностороннее хеширование). Попытка rainbow table attack неэффективна из-за соли, встроенной в bcrypt.

- 2 критических уязвимости (открытый SSH, отсутствие MFA)
- 4 высокоуровневые уязвимости (три зашифрованных EBS тома, устаревший снимок состояния)
- 5 среднеуровневых проблем (два неиспользуемых EIP по $3.60/месяц каждый, неприменённый том EBS, неиспользуемая группа безопасности, EC2 инстанс с CPU < 5%)
- 1 низкоуровневая проблема

Доля неэффективных расходов в общем бюджете: 16.6% ($833/месяц).

Индекс безопасности (Health Score): 35 пунктов из 100 возможных.

**Процесс оптимизации:**

На первом этапе разработано руководство по исправлению критических уязвимостей с использованием интеллектуального модуля (AI Advisor). Предложенные меры:

1. Включение многофакторной аутентификации (MFA) на root-аккаунте (5 минут рабочего времени)
2. Ограничение доступа SSH на IP корпоративного офиса вместо всемирного доступа (10 минут)
3. Удаление двух неиспользуемых EIP (2 минуты, экономия $7.20/месяц)
4. Удаление устаревшего снимка состояния диска (2 минуты, экономия $15/месяц)

Общее время внедрения мер: менее 1 часа рабочего времени.

**Результаты после оптимизации:**

- Общие расходы: $4,932/месяц (было $5,045/месяц) — снижение на $113/месяц (2.2%)
- Потраченные ресурсы: $12/месяц (было $833/месяц) — 98.6% сокращение неэффективных расходов
- Индекс безопасности: 62 пункта (было 35) — улучшение на 27 пунктов
- Оставшиеся уязвимости: 7 (было 12), включая 0 критических

**Долгосрочная оптимизация (недели 2–4):**

Углубленный анализ семидневной истории использования ресурсов с привлечением AI показал дополнительные возможности оптимизации:

- 3 EC2 инстанса с CPU utilization 5–10%: потенциал downsize с t3.large на t3.small (-$45/месяц)
- 1 неиспользуемый EC2 инстанс: удаление (-$30/месяц)
- 5 EC2 инстансов с постоянным 70%+ utilization: переход на Reserved Instances со скидкой 40% (-$200/месяц)

Общее потенциальное сокращение расходов: $275/месяц = $3,300/год (5.5% от начальной суммы).

**Финальное состояние (месяц 1):**

- Общие расходы: $3,755/месяц (было $5,045) — 25.6% экономия
- Потраченные ресурсы: менее $10/месяц — практически полная ликвидация неэффективных расходов
- Индекс безопасности: 92 пункта (было 35) — максимальное улучшение
- Оставшиеся уязвимости: 1 низкоуровневая

### 3.5.2 Анализ эффективности для растущего предприятия

Второй сценарий рассматривает предприятие с ежемесячным AWS spend около $15,000 и сложной инфраструктурой (200 EC2, 150 EBS, 25 EIP, 40 Security Groups, несколько RDS databases).

**Исходная проблема:**

DevOps-команда не имеет полной видимости распределения расходов по компонентам инфраструктуры. Разрыв между бюджетом и фактическими расходами составляет $2,500/месяц (14.7%).

**Применение системы:**

Первичное сканирование выявило детальное распределение: Production ($7,000), Staging ($3,000), Development ($2,500), Unused ($2,500).

Анализ с использованием AI выявил потери:

- Development-инстансы не отключаются ночью и в выходные (потеря ~$800/месяц)
- Старые EBS-снимки состояния от года назад занимают место (потеря ~300/месяц)
- Reserved Instances неправильно распределены между инстансами
- Несколько RDS databases в режиме Multi-AZ для Development (потеря ~400/месяц)

**Процесс оптимизации (недели 2–4):**

Внедрены следующие политики:

- Auto-shutdown Development-инстансов в 6 PM по UTC, автозапуск в 8 AM
- Ротация EBS-снимков (сохранение максимум 3 месяцев истории)
- Переконфигурация Reserved Instances для Production-нагрузок
- Переход Development RDS на Single-AZ (ликвидация Multi-AZ для dev)

**Результаты:**

- Месячная экономия: $1,500–$1,800 (10–12% от исходного budget)
- Год: $18,000–$21,600 экономии
- Побочный эффект: улучшена видимость инфраструктуры, команда начала применять best practices

### 3.5.3 Анализ эффективности для крупного предприятия

Третий сценарий рассматривает корпоративное окружение с 100+ AWS-аккаунтами, 1000+ EC2-инстансов, мультирегиональной инфраструктурой и месячным spend $60,000+.

**Исходная сложность:**

Различные DevOps-команды работают независимо, что приводит к дублированию ресурсов, конфликтам за недостаточно распределённые ресурсы и отсутствию единой политики оптимизации.

**Развёртывание системы:**

Внедрена многоаккаунтная конфигурация AWS Optimizer:

- Каждому team назначена собственная instance системы с доступом только к своим AWS-аккаунтам
- Ежедневные сканирования с автоматическими алертами в Slack
- Централизованный дашборд для финансовых аналитиков (агрегированные метрики всех teams)

**Результаты через 3 месяца:**

- Общая экономия: $7,200/месяц (12% от исходного бюджета)
- Годовая экономия: $86,400
- Улучшение Health Score: в среднем на 40 пунктов за счёт стандартизации политик безопасности
- Сокращение security incidents: примерно на 30% благодаря систематическому выявлению и исправлению уязвимостей
