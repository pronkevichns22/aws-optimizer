# ГЛАВА 2. АРХИТЕКТУРА СИСТЕМЫ И ТЕХНОЛОГИЧЕСКИЙ СТЕК

## 2.1 Обзор архитектуры системы

Разработка комплексной системы для аудита и оптимизации облачной инфраструктуры требует тщательного выбора архитектурной парадигмы, которая бы обеспечила масштабируемость, модульность и возможность эволюции системы. В современной разработке программного обеспечения существует несколько основных архитектурных подходов, каждый из которых обладает своими преимуществами и недостатками.

Монолитная архитектура представляет собой подход, при котором все компоненты системы собраны в единое целое и развертываются как один экземпляр приложения. Преимущества монолитной архитектуры включают простоту начальной разработки, легкость отладки и хорошую производительность при малом масштабе. Однако по мере роста системы монолитная архитектура становится все менее практичной: изменение одного компонента требует переборки и переразвертывания всей системы, а различные части приложения могут использовать несовместимые версии зависимостей.

Микросервисная архитектура предлагает альтернативный подход, разбивая систему на множество небольших, независимо развертываемых сервисов, каждый из которых выполняет одну отдельную бизнес-функцию. Каждый микросервис имеет свою собственную базу данных, может быть написан на различных языках программирования и развертывается независимо. Преимущества микросервисной архитектуры включают гибкость, масштабируемость отдельных компонентов и возможность независимого развития команд. Однако микросервисная архитектура значительно усложняет систему в целом, требует продвинутых методов организации (контейнеризация, оркестрация), и может привести к снижению производительности из-за сетевых задержек при общении между сервисами.

Для разработки данной системы была выбрана модифицированная монолитная архитектура с элементами модульности. Решение было принято на основе следующих соображений. Во-первых, область применения системы определена и достаточно хорошо структурирована: система должна выполнять три основные функции — сбор данных из облачной инфраструктуры, анализ конфигураций и финансовых метрик, а также формирование рекомендаций. Во-вторых, ожидаемый масштаб развертывания позволяет избежать сложностей микросервисной архитектуры. В-третьих, монолитный подход позволяет начать разработку быстро и демонстрировать рабочие прототипы в короткие сроки.

Однако система проектируется с учетом возможного преобразования в микросервисную архитектуру в будущем. Внутри монолитного приложения реализована трехслойная архитектура, четко разделяющая ответственность между компонентами. Presentation Layer отвечает за взаимодействие с конечным пользователем через веб-интерфейс. Application Layer содержит бизнес-логику системы, включая модули анализа безопасности, расчета финансовых метрик и интеграции с AI-консультантом. Data Persistence Layer управляет хранилищем данных и абстрагирует детали работы с базой данных от верхних слоев. Такая модульная структура позволяет разработчикам работать параллельно над различными компонентами, облегчает тестирование и обслуживание системы.

Другим важным архитектурным решением является выбор паттерна Model-View-Controller для фронтенд-компонента. Этот паттерн обеспечивает четкое разделение между представлением данных, логикой приложения и источниками данных. Model содержит состояние приложения и бизнес-логику, View отвечает за отображение информации пользователю, а Controller обрабатывает пользовательские действия и обновляет Model. Использование MVC паттерна упрощает разработку сложных пользовательских интерфейсов и облегчает их тестирование.

Выбор технологического стека является одним из критических решений при разработке системы. Он определяет производительность, масштабируемость, поддерживаемость и скорость разработки. Для разработки данной системы был выбран следующий технологический стек: React 19 с TypeScript для фронтенда, Express.js с Node.js для бэкенда и MongoDB для хранилища данных. Для разработки пользовательского интерфейса была выбрана библиотека React версии 19. React является одной из наиболее популярных и зрелых библиотек для создания интерактивных веб-интерфейсов. React использует декларативный подход к описанию пользовательского интерфейса, при котором разработчик описывает, как должен выглядеть интерфейс в любой момент времени, а сама библиотека управляет изменениями виртуального DOM и синхронизирует их с реальным DOM. Это значительно упрощает разработку сложных интерфейсов и снижает вероятность ошибок.

React имеет огромную экосистему инструментов и библиотек, включая инструменты для управления состоянием приложения, маршрутизацией, валидацией форм и тестированием. Большое сообщество разработчиков обеспечивает постоянное развитие экосистемы и доступность огромного количества примеров кода и документации. Производительность React при работе с большим количеством данных остается высокой благодаря оптимизациям виртуального DOM и поддержке асинхронной обработки. TypeScript был выбран как язык разработки фронтенда вместо обычного JavaScript. TypeScript добавляет систему статической типизации к JavaScript, позволяя выявлять ошибки на этапе разработки, а не во время выполнения. При работе с большим кодовым базом типизация значительно облегчает поддержку кода и снижает вероятность ошибок при рефакторинге. TypeScript также улучшает автозавершение кода в IDE и облегчает документирование кода через типы.

Для разработки серверной части приложения была выбрана платформа Node.js с фреймворком Express.js. Node.js является платформой для выполнения JavaScript на сервере, что позволяет использовать единый язык программирования как для фронтенда, так и для бэкенда. Это упрощает обмен кодом между частями приложения и снижает кривую обучения для разработчиков, знакомых с JavaScript. Node.js построен на основе асинхронной модели обработки событий, что делает его чрезвычайно эффективным для работы с вводом-выводом, характерным для веб-приложений. Приложение на Node.js может обрабатывать тысячи одновременных соединений в режиме высокой нагрузки без необходимости в многопоточности. Это позволяет достичь высокую пропускную способность с минимальными затратами на ресурсы.

Express.js является минималистичным фреймворком для построения веб-приложений на Node.js. Express предоставляет удобные механизмы для определения маршрутов приложения, обработки HTTP-запросов, работы с middleware, и управления сессиями. Несмотря на свою простоту, Express обладает достаточной функциональностью для построения сложных приложений и имеет обширную экосистему расширений.

Для хранилища данных была выбрана база данных MongoDB. MongoDB является документоориентированной базой данных, которая хранит данные в виде документов JSON. Это позволяет более естественным образом представлять сложные структуры данных по сравнению с реляционными базами данных. Схема документов может изменяться во время разработки без необходимости миграции базы данных, что ускоряет итеративную разработку. MongoDB обладает встроенной поддержкой горизонтального масштабирования через шардирование, позволяя системе работать с большими объемами данных, распределенными по нескольким серверам. Язык запросов MongoDB интуитивно понятен и достаточно мощен для выполнения сложных запросов и агрегаций данных. MongoDB легко интегрируется с приложениями на Node.js благодаря наличию полнофункциональных драйверов и объектно-документных мапперов, таких как Mongoose.

Для интеграции с искусственным интеллектом была выбрана платформа Groq Llama. Groq предоставляет облачный API для доступа к большим языковым моделям, включая LLama-2 и более новые версии. Groq известна высокой скоростью обработки запросов к LLM благодаря специализированному аппаратному обеспечению. Выбор Groq вместо других провайдеров LLM (например, OpenAI) был сделан на основе баланса между стоимостью, скоростью обработки и доступностью API.

Такой технологический стек обеспечивает оптимальный баланс между производительностью, разработкой, поддерживаемостью и стоимостью. Все компоненты стека являются зрелыми, хорошо документированными, имеют большие сообщества разработчиков и поддерживаются на долгосрочной основе. Выбранные технологии позволяют разработчикам быстро создавать новые функции и легко масштабировать систему при необходимости.

## 2.2 Архитектура фронтенда и пользовательский интерфейс

Разработанная система состоит из нескольких ключевых компонентов, каждый из которых отвечает за определенный набор функций. Компоненты организованы в соответствии с трехслойной архитектурой и взаимодействуют между собой через четко определенные интерфейсы. Presentation Layer содержит веб-интерфейс приложения, разработанный на React. Этот слой отвечает за отображение информации пользователю и обработку его действий. 

Веб-интерфейс состоит из нескольких основных страниц. Dashboard представляет собой главную страницу приложения, на которой отображается общее состояние облачной инфраструктуры, ключевые метрики безопасности и финансовые показатели. Страница отображает четыре основные метрики: общие расходы на облачные сервисы, потраченные впустую ресурсы, количество обнаруженных конфигурационных проблем и количество критичных проблем безопасности. На странице также присутствует график трендов расходов за последние семь дней и распределение затрат по типам сервисов. Кнопка "Run Scan" позволяет пользователю инициировать новое сканирование своей AWS инфраструктуры.

Scan Results Page показывает подробные результаты последнего сканирования, включая выявленные проблемы, их классификацию по типам угроз и рекомендации по исправлению. На этой странице выводится список всех обнаруженных проблем со степенью серьезности (CRITICAL, HIGH, MEDIUM, LOW), типом проблемы (безопасность, финансы или best practice), описанием проблемы и рекомендацией по исправлению. Пользователь может фильтровать проблемы по типу и серьезности для удобства просмотра.

Financial Dashboard отображает финансовые метрики использования ресурсов, включая идентификацию неиспользуемых ресурсов и потенциальную экономию. На этой странице выводится список ресурсов, которые используются неэффективно или вообще не используются, с расчетом ежемесячной экономии при удалении или оптимизации каждого ресурса. Страница помогает компаниям идентифицировать утечки бюджета и принимать решения по оптимизации расходов.

AI Advisor является интерактивным компонентом, позволяющим пользователям задавать вопросы о выявленных проблемах и получать контекстные рекомендации на основе искусственного интеллекта. Компонент отображается в виде модального окна с историей сообщений. Пользователь может вводить вопросы на естественном языке, а система будет генерировать ответы, объясняя проблему, её потенциальные последствия и пошаговые инструкции по исправлению.

## 2.3 Архитектура бэкенда и бизнес-логика

Application Layer содержит основную бизнес-логику системы и состоит из нескольких основных компонентов. AWS Integration Module отвечает за подключение к облачной инфраструктуре AWS через AWS SDK, сбор данных о ресурсах, конфигурациях и метриках использования. Этот модуль абстрагирует детали взаимодействия с AWS API и предоставляет унифицированный интерфейс для остальной системы. Security Analysis Module реализует алгоритмы проверки конфигураций на соответствие стандартам безопасности, включая более двадцати автоматизированных правил проверки на основе CIS AWS Foundations Benchmark и AWS Well-Architected Framework. Этот модуль выявляет конфигурационные уязвимости и классифицирует их по уровням серьезности.

FinOps Analysis Module занимается анализом использования ресурсов и расчетом финансовых метрик. Модуль идентифицирует неиспользуемые ресурсы, рассчитывает потенциальную экономию при их удалении, анализирует тренды использования и прогнозирует будущие расходы. Recommendations Engine использует результаты работы других модулей для формирования практических рекомендаций по исправлению выявленных проблем и оптимизации расходов. Этот компонент интегрирует результаты анализа безопасности и финансового анализа для создания приоритизированного списка действий.

AI Advisor Module интегрирует большую языковую модель Groq Llama для формирования интерактивных рекомендаций. Этот модуль получает контекст из других компонентов системы (выявленные проблемы, финансовые данные, архитектура инфраструктуры) и генерирует контекстные ответы на вопросы пользователя. AI Advisor может объяснить, почему обнаруженная проблема является критичной, какие последствия она может иметь, и как конкретно ее исправить в контексте конкретной инфраструктуры организации.

Data Persistence Layer управляет хранилищем данных и состоит из MongoDB базы данных и соответствующего драйвера для доступа к данным. В MongoDB хранятся результаты сканирований, история проблем безопасности, финансовые метрики, пользовательские конфигурации и кэшированные данные из AWS API. Структура хранилища оптимизирована для быстрого выполнения типичных запросов и поддержки аналитических операций.

Взаимодействие между компонентами происходит через API endpoints, определяемые в Application Layer. Когда пользователь инициирует сканирование инфраструктуры через Presentation Layer, запрос передается в AWS Integration Module, который собирает данные о ресурсах. Собранные данные передаются в Security Analysis Module для проверки конфигураций и в FinOps Analysis Module для анализа расходов. Результаты анализа агрегируются Recommendations Engine и сохраняются в Data Persistence Layer. Затем результаты отображаются пользователю через Dashboard на Presentation Layer. При обращении пользователя к AI Advisor контекст выявленных проблем передается в AI Advisor Module, который формирует запрос к модели Groq Llama с контекстом о проблеме, конфигурации инфраструктуры и лучших практиках. Модель генерирует рекомендацию, которая отображается пользователю в интерактивном формате с возможностью задать дополнительные вопросы.

Коммуникация между фронтенд и бэкенд компонентами осуществляется через RESTful API endpoints, определенные в Express.js приложении. API использует стандартные HTTP методы (GET, POST, PUT, DELETE) и возвращает JSON формат для всех ответов, что обеспечивает универсальную совместимость с различными клиентами. Все запросы требуют авторизации через Bearer токен, передаваемый в заголовке Authorization. Сервер проверяет валидность токена перед обработкой любого запроса и возвращает ошибку 401 Unauthorized в случае отсутствия или истечения срока действия токена.

Выбор Express.js был сделан после анализа альтернативных фреймворков для Node.js. В сравнении с другими решениями Express.js демонстрирует оптимальный баланс между простотой использования и функциональностью. Fastify предлагает несколько более высокую производительность, но меньшую экосистему и меньше готовых решений для типичных задач. Koa известна большей гибкостью, но требует большего объема кода для реализации стандартных функций. Express.js, будучи де-факто стандартом для Node.js приложений, позволил команде сосредоточиться на бизнес-логике вместо борьбы с фреймворком.

Аналогично, MongoDB была выбрана как оптимальное хранилище благодаря гибкости схемы, что критически важно при разработке сложных систем с быстро эволюционирующей структурой данных. --- Endpoint POST /auth/register принимает адрес электронной почты и пароль, проверяет уникальность электронной почты в системе, хеширует пароль с использованием алгоритма bcrypt и создает новую запись пользователя в MongoDB. При успешной регистрации сервер возвращает объект пользователя с уникальным идентификатором и JWT токеном. Endpoint POST /auth/login принимает учетные данные пользователя, выполняет поиск пользователя в базе данных по электронной почте, проверяет корректность пароля, и в случае успешной проверки генерирует session token с временем жизни 24 часа и возвращает его клиенту.

Endpoints управления облачными аккаунтами позволяют пользователям подключать и управлять AWS аккаунтами. Endpoint POST /accounts/connect принимает AWS аккаунт ID и учетные данные (access key и secret key или временные токены), проверяет возможность подключения к AWS API с предоставленными учетными данными, и сохраняет информацию о подключенном аккаунте в MongoDB. Учетные данные AWS хранятся в зашифрованном виде с использованием алгоритма AES-256.


## 2.4 Модель данных и хранилище

Система использует MongoDB для хранения данных благодаря гибкости схемы и возможности хранения сложных вложенных документов, которые соответствуют иерархической структуре облачной инфраструктуры AWS. Архитектура базы данных состоит из нескольких ключевых коллекций, каждая из которых хранит специфический тип информации. Коллекция users хранит информацию о пользователях системы, включая адрес электронной почты, хешированный пароль, хеш токена сеанса, дату последнего входа и список подключенных облачных аккаунтов AWS. Каждый документ пользователя содержит массив идентификаторов AWS аккаунтов, к которым у пользователя есть доступ, что позволяет поддерживать многоаккаунтные сценарии.

Коллекция scans хранит информацию о каждом выполненном сканировании инфраструктуры. Каждый документ содержит временную метку начала и завершения сканирования, статус сканирования, список обнаруженных ресурсов, время выполнения и статистику обнаруженных проблем. Документы в этой коллекции содержат вложенные объекты с подробным описанием каждого ресурса EC2, EBS, S3, RDS, IAM, VPC и других, включая все их параметры конфигурации. Это позволяет системе восстанавливать состояние инфраструктуры в момент времени конкретного сканирования.

Коллекция findings содержит выявленные проблемы безопасности и финансовые рекомендации. Каждый документ проблемы содержит: идентификатор затронутого ресурса, тип проблемы, категорию (security, finops, best-practice), уровень серьезности (Critical, High, Medium, Low), детальное описание проблемы, рекомендацию по исправлению, статус решения (open, resolved, ignored), дату обнаружения и дату последнего обновления статуса. Документы также содержат оценку потенциальной экономии в долларах для финансовых рекомендаций и информацию о соответствующих стандартах безопасности (CIS, AWS Well-Architected, NIST, ISO).

> **⚠️ [STATUS: PARTIAL] Поле статуса (open, resolved, ignored) реализовано. Поля для snoozeUntil, acknowledgedBy, resolution notes НЕ реализованы. История изменений статуса (audit trail) также отсутствует.**

Коллекция recommendations агрегирует все рекомендации в единую приоритизированную очередь. Документы содержат рекомендацию, ее приоритет (основанный на комбинации серьезности и потенциальной экономии), список связанных проблем, предполагаемое время реализации, уровень сложности исправления и шаги для исправления. Коллекция audit_logs ведет логирование всех действий в системе, включая: вход пользователей, инициирование сканирований, запуск автоматизированных исправлений, изменение конфигураций, экспорты отчетов. Коллекция cache хранит кэшированные данные из AWS API для снижения нагрузки на API и ускорения повторных запросов.

Индексирование в MongoDB оптимизировано для типичных запросов системы. Индекс на user_id в коллекции scans позволяет быстро получать все сканирования конкретного пользователя. Индекс на status в коллекции findings позволяет быстро отфильтровать открытые проблемы. Индекс на timestamp в коллекции scans позволяет быстро получать последние сканирования. Составной индекс на resource_id и type в коллекции findings позволяет быстро найти все проблемы конкретного ресурса определенного типа. Такие индексы критичны для производительности при работе с большими объемами данных.

Документ в коллекции findings содержит: уникальный идентификатор проблемы, ID сканирования, ID ресурса и его тип, категорию проблемы (безопасность, финансы, лучшие практики), уровень серьезности, детальное описание, рекомендацию по исправлению, текущий статус, соответствующие стандарты безопасности (CIS, AWS Well-Architected) и дату обнаружения. Вот пример структуры документа проблемы:

```json
{
  "resourceId": "sg-0123456789abcdef",
  "type": "security",
  "severity": "CRITICAL",
  "title": "SSH exposed to the internet",
  "status": "open",
  "discoveredAt": "2024-05-13T10:30:00Z"
}
```

Документ в коллекции recommendations агрегирует рекомендации в приоритизированном порядке, содержа: приоритет, описание действия, уровень сложности исправления, предполагаемое время реализации, потенциальную экономию в долларах, пошаговые инструкции по исправлению и ссылки на документацию AWS.

Безопасность системы строится на нескольких уровнях защиты. На уровне аутентификации система использует стандартный механизм логина и пароля с дополнительной защитой: пароли хешируются с использованием алгоритма bcrypt с коэффициентом раунда 12, что обеспечивает достаточную стоимость вычислений для предотвращения атак перебора. После успешной аутентификации пользователю выдается токен сеанса (session token), который используется для всех последующих запросов. Токены хранятся в MongoDB с временем жизни, по истечении которого токен автоматически инвалидируется. На уровне транспорта все соединения используют HTTPS для шифрования данных при передаче между клиентом и сервером. На уровне хранения учетные данные AWS не хранятся в системе напрямую, вместо этого используется механизм временных токенов или IAM ролей. Пользователи предоставляют учетные данные AWS через безопасный HTTPS канал во время инициализации подключения, после чего система использует эти учетные данные только для запросов к AWS API и не сохраняет их локально.

На уровне приложения реализована проверка прав доступа перед любой операцией. Пользователь может получить доступ только к сканированиям и рекомендациям, относящимся к его аккаунтам AWS. Система отслеживает, какой AWS аккаунт подключен конкретным пользователем и гарантирует, что пользователь может видеть только данные этого аккаунта. Если пользователь пытается получить доступ к данным, не принадлежащим его аккаунтам, система возвращает ошибку авторизации. В базе данных все документы содержат информацию о владельце (user_id и account_id), что позволяет системе проверять права доступа при каждом запросе.

## 2.5 Интеграция с AI (Groq Llama)

Для интеграции с искусственным интеллектом система использует облачный API платформы Groq, которая предоставляет доступ к большим языковым моделям, включая Llama 3.1 8B. Groq известна высокой скоростью обработки запросов к LLM благодаря специализированному аппаратному обеспечению. Выбор Groq вместо других провайдеров LLM (например, OpenAI) был сделан на основе баланса между стоимостью, скоростью обработки и доступностью API.

AI Advisor Module интегрирует эту модель для формирования интерактивных рекомендаций. Этот модуль получает контекст из других компонентов системы (выявленные проблемы, финансовые данные, архитектура инфраструктуры) и генерирует контекстные ответы на вопросы пользователя. AI Advisor может объяснить, почему обнаруженная проблема является критичной, какие последствия она может иметь, и как конкретно ее исправить в контексте конкретной инфраструктуры организации. Ответ может содержать примеры команд AWS CLI для исправления, примеры кода Infrastructure as Code (Terraform, CloudFormation) и лучшие практики для предотвращения аналогичных проблем в будущем.

При активации AI Advisor система подготавливает контекст, содержащий: полное описание выявленной проблемы, конфигурацию затронутого ресурса, архитектурный контекст (какие другие ресурсы зависят от этого ресурса, как изменение влияет на систему), рекомендуемое действие, потенциальные риски. Этот контекст отправляется в Groq Llama API вместе с вопросом пользователя:

```typescript
const response = await groq.chat.completions.create({
  model: 'llama-3.1-8b-instant',
  messages: [
    { role: 'system', content: 'AWS security expert helping fix infrastructure' },
    { role: 'user', content: problemDescription + userQuestion }
  ]
});
```

Модель анализирует контекст и генерирует ответ, который объясняет проблему в доступной форме, предоставляет пошаговые инструкции по исправлению (включая AWS CLI команды), описывает потенциальные риски и предлагает лучшие практики для предотвращения аналогичных проблем в будущем.

Фронтенд приложение использует эти API endpoints через асинхронные HTTP запросы, инициируемые при пользовательских действиях. Компонент Dashboard загружает данные с endpoint /scans/latest при монтировании компонента для отображения результатов последнего сканирования. При клике пользователя на кнопку "Запустить сканирование" фронтенд отправляет POST запрос на /scans/run и периодически опрашивает для отслеживания прогресса. Компонент Findings List загружает проблемы с /findings и обновляет отображение при изменении фильтров. Компонент AI Advisor отправляет запросы на /ai/ask и отображает ответ LLM в интерактивном формате. Все запросы включают Authorization заголовок с сохраненным токеном пользователя. При возврате ошибки 401 фронтенд автоматически перенаправляет пользователя на страницу входа.

Интеграция системы обеспечивает полный цикл обработки: от сбора данных об инфраструктуре AWS через автоматизированные проверки безопасности и финансовый анализ до формирования рекомендаций и интерактивной консультации через AI. Такая архитектура позволяет пользователям получить полное понимание состояния своей облачной инфраструктуры, выявить проблемы безопасности и финансовые потери, а также получить практические рекомендации по их исправлению с помощью искусственного интеллекта.

## Выводы по второй главе

Выбранная архитектура и технологический стек обеспечивают оптимальный баланс между функциональностью, масштабируемостью и скоростью разработки. Модифицированная монолитная архитектура с трехслойной организацией компонентов позволяет быстро разрабатывать и тестировать функции, при этом сохраняя модульность, необходимую для будущих расширений и эволюции системы. React 19 с TypeScript обеспечивает надежный, типизированный фронтенд с отличной производительностью при работе с большими объемами данных. Express.js остается минималистичным, но мощным выбором для бэкенда, позволяя эффективно обрабатывать большое количество одновременных запросов. MongoDB предоставляет гибкость схемы, необходимую для быстрой итеративной разработки в условиях меняющихся требований, и встроенную поддержку горизонтального масштабирования.

Интеграция с Groq Llama обеспечивает искусственный интеллект, необходимый для предоставления контекстных, практических рекомендаций пользователям. Многоуровневая система безопасности, включающая шифрование, аутентификацию, авторизацию и аудирование, защищает данные пользователей и инфраструктуры AWS. Выбранные технологии являются зрелыми, хорошо документированными и активно поддерживаемыми сообществом разработчиков, что обеспечивает надежность и долгосрочную жизнеспособность системы. Такая архитектурная и технологическая основа готова к практической реализации всех функциональных требований системы для аудита и оптимизации облачной инфраструктуры AWS.
