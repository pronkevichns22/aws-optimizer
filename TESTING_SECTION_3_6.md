# 3.6 Верификация и комплексное тестирование программного комплекса

Для подтверждения корректности работы реализованных алгоритмов и обеспечения высокого уровня отказоустойчивости системы был проведен цикл автоматизированного тестирования. Процесс верификации охватывал проверку математических моделей финансовых расчетов, криптографических механизмов защиты и логики выявления конфигурационных уязвимостей.

## 3.6.1 Архитектура многоуровневого процесса сканирования

Процесс сканирования в разработанном программном комплексе реализован как многоуровневый конвейер (Pipeline), состоящий из этапа инвентаризации активов, глубокого комплаенс-аудита и специализированной алгоритмической проверки. Такое разделение позволяет достичь максимальной полноты данных при сохранении высокой скорости обработки.

### Этап 1: Инвентаризация ресурсов (Resource Discovery)

Первым этапом конвейера является сбор первичных конфигурационных данных. В отличие от простых сканеров, разработанный комплекс не просто ищет уязвимости, а формирует полную карту ресурсов облака.

Для взаимодействия с облачной средой применяется библиотека AWS SDK v3. Программная логика инициирует параллельный опрос программных интерфейсов (API) через механизм Promise.all():

```typescript
const [instances, volumes, elasticIPs, securityGroups] = await Promise.all([
    ec2.send(new DescribeInstancesCommand({})),     // Виртуальные серверы
    ec2.send(new DescribeVolumesCommand({})),       // Дисковые тома
    ec2.send(new DescribeAddressesCommand({})),     // Статические IP
    ec2.send(new DescribeSecurityGroupsCommand({})) // Сетевые экраны
]);
```

Полученные данные нормализуются и подготавливаются для последующих этапов анализа. Наличие полных метаданных ресурсов позволяет системе в дальнейшем рассчитывать финансовые метрики и формировать полный отчет о конфигурации.

### Этап 2: Глубокий комплаенс-аудит (Prowler Integration)

Вторым уровнем проверки является интеграция с отраслевым стандартом аудита безопасности – утилитой Prowler. Данный модуль отвечает за проверку инфраструктуры на соответствие жестким нормативным требованиям (CIS AWS Foundations Benchmark).

Механика интеграции реализована следующим образом:

- **Системный вызов**: Серверная часть инициирует выполнение процесса Prowler в изолированной среде Python.
- **Параметризация**: Запрос ограничивается целевым регионом и набором учетных данных, предоставленных пользователем.
- **Парсинг результатов**: Результаты работы Prowler, возвращаемые в формате JSON, проходят через специализированный обработчик (Parser), который извлекает критичность (Severity), код стандарта (например, CIS 2.1) и ссылку на техническую документацию по устранению уязвимости.

Этот этап позволяет выявить сложные конфигурационные ошибки в сервисах IAM, S3 и CloudTrail, которые требуют глубокого анализа политик доступа.

В ходе выполнения аудита в консоли сервера (Backend) отображается процесс прохождения проверок по различным группам сервисов AWS (IAM, EC2, RDS, S3, CloudTrail, VPC). На рисунке 3.4 представлен фрагмент вывода системы в процессе сканирования тестовой инфраструктуры.

### Этап 3: Встроенные правила анализа (Custom Rules Engine)

Третьим уровнем анализа является запуск собственного ядра правил, реализованного на языке TypeScript. В отличие от Prowler, который сфокусирован на ИБ-комплаенсе, встроенный движок решает задачи финансовой оптимизации (FinOps) и специфических проверок безопасности.

Встроенное ядро правил выполняет следующие проверки:

**Анализ конфигурации безопасности:**
- Оценка правил Security Groups на предмет открытых портов SSH (22) и RDP (3389)
- Выявление разрешительных правил для критических портов баз данных (3306, 5432, 27017)
- Идентификация неиспользуемых Security Groups, которые не содержат привязанных ресурсов

**Анализ финансовой оптимизации:**
- Расчет затрат на EBS тома и выявление нешифрованных дисков
- Анализ использования Elastic IP адресов и определение неассоциированных адресов
- Оценка состояния EC2 инстансов (running/stopped) и потенциала экономии

## 3.6.2 Методология автоматизированного тестирования

В качестве основного инструмента тестирования был выбран фреймворк Jest, обеспечивающий высокую скорость выполнения тестов и детальную отчетность по покрытию кода. Система тестирования включает в себя 243 функциональных теста, сгруппированных в 12 тестовых наборов (Test Suites).

Процесс верификации был разделен на следующие ключевые домены:

### **Security Rules Engine (Ядро безопасности):**

- **Rule 1.1 (SSH Exposure)**: проверка корректности детекции открытых портов 22 для CIDR 0.0.0.0/0 и отсутствие ложноположительных срабатываний для ограниченных диапазонов.

- **Rule 2.1 (EBS Encryption)**: верификация алгоритма поиска нешифрованных томов и корректность подсчета зашифрованных дисковых ресурсов.

- **Rule 4.1 (IAM Best Practices)**: автоматическая идентификация учетных записей без активированной многофакторной аутентификации (MFA) и использования root-аккаунта.

### **FinOps Calculator (Финансовый калькулятор):**

- Расчет стоимости EC2 инстансов в зависимости от их типа (t3.medium, p3.8xlarge) и текущего состояния (running/stopped).

- Агрегация затрат по различным категориям ресурсов и корректная обработка дисковых томов большого объема.

### **Authentication & Cryptography (Безопасность доступа):**

- **Password Hashing**: проверка работы алгоритма bcrypt, включая верификацию корректных паролей и отказ в доступе при несовпадении хеша.

- **JWT Tokens**: верификация механизмов создания подписи, проверки срока жизни токена и обработки попыток использования поддельных маркеров доступа.

- **AWS Credentials Validation**: проверка структурной корректности вводимых ключей Access Key и Secret Key.

## 3.6.4 Итоги верификации

Успешное прохождение 243 тестов подтверждает стабильность алгоритмической базы комплекса CloudOpti. Достижение 100% покрытия кода в расчетных и защитных модулях гарантирует, что финансовые отчеты и механизмы авторизации работают безошибочно. Разработанный набор тестов позволяет минимизировать риски внесения регрессионных дефектов при масштабировании системы для работы в промышленной среде AWS.
