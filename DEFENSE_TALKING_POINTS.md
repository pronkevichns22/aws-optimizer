## Основная часть

### Слайд 1: Проблема (2-3 минуты)

**Сказать:**

> "Сегодня облачные вычисления — это ядро цифровой трансформации. AWS владеет 32% рынка.
> 
> Однако существует парадокс: 60% успешных кибератак на облачные инфраструктуры происходят НЕ из-за 
> эксплуатации уязвимостей в коде AWS, а из-за НЕПРАВИЛЬНОЙ КОНФИГУРАЦИИ пользователями.
> 
> Компании теряют:
> - В безопасности: инциденты обнаруживаются за 200+ дней (вместо часов)
> - В деньгах: 30% облачного бюджета на неиспользуемые ресурсы
> 
> На рынке существуют решения, но:
> - AWS Trusted Advisor: дорогой ($100-10k/месяц), стандартный интерфейс
> - Prowler: бесплатный, но требует компетенций DevOps инженера
> - Cloud Custodian: очень мощный, но тоже требует навыков программирования
> 
> **Никто не объединил CSPM (безопасность) + FinOps (финансы) + AI (консультирование) в одном месте.**"

**Показать диаграмму 3-слойной архитектуры и сказать:**

> "Я разработал AWS Optimizer — веб-платформу с тремя слоями:
> 
> **Слой 1 (Frontend):** React 19 + TypeScript
> - 7 страниц: Логин, Dashboard, Resources, Security, Settings
> - Красивый UI для непрограммистов
> 
> **Слой 2 (Backend):** Express.js + Node.js + TypeScript
> - AWS SDK для сканирования инфраструктуры
> - Rules Engine с 20+ правилами проверки
> - Groq LLM интеграция для консультирования
> 
> **Слой 3 (Data):** MongoDB
> - Хранение результатов сканирований
> - История диалогов с AI
> - Безопасное хранение AWS credentials (AES-256 шифрование)
> 
> **Ключевая инновация:** Все 4 AWS API запроса (EC2, EBS, Elastic IPs, Security Groups) 
> происходят **параллельно**, а не последовательно.
> Это ускоряет сканирование в 4 раза: с 12.8 сек до 3.2 сек."

## Ответы на возможные вопросы

### Вопрос 1: "Это же просто обертка над AWS SDK?"

**Правильный ответ:**

> "Технически да, это использует AWS SDK. Но давайте посмотрим, что я реально разработал:
> 
> 1. **Архитектура:** 3-слойная система с модульностью (можно переделать в микросервисы)
> 2. **Rules Engine:** 20 встроенных правил, которые я сам разработал на основе анализа CIS Benchmark
> 3. **Параллельная обработка:** Ускорил сканирование в 4 раза за счет параллельных запросов
> 4. **AI интеграция:** Детерминированное консультирование (не генеративное)
> 5. **UI/UX:** Полнофункциональный веб-интерфейс для непрограммистов
> 6. **Безопасность:** Шифрование credentials, JWT токены, bcrypt хеширование
> 
> Это не 'обертка' — это полнофункциональное инженерное решение реальной проблемы."

### Вопрос 2: "Почему вы не использовали готовые правила от Prowler?"

**Правильный ответ:**

> "Я рассмотрел две стратегии:
> 
> **Вариант 1 (Prowler интеграция):**
> - Плюс: 400+ готовых правил
> - Минус: Требует установки Python и CLI
> - Минус: Сложнее интегрировать в веб-интерфейс
> - Минус: Медленнее (CLI запуск ~ 2-10 минут)
> 
> **Вариант 2 (встроенные правила):**
> - Плюс: Быстро (< 15 сек)
> - Плюс: Работает в веб-приложении
> - Плюс: Я их контролирую и могу адаптировать
> - Минус: Покрыватый только 25% из 80+ CIS рекомендаций
> 
> Я выбрал Вариант 2, потому что:
> 1. Это достаточно для 80% практических сценариев
> 2. Система может дополнительно запускать Prowler, если инженер установит его
> 3. Это показывает, что я разбираюсь в CSPM, а не просто использую чужой код"

### Вопрос 3: "Почему Groq, а не OpenAI/Claude?"

**Правильный ответ:**

> "Я выбрал Groq по трем причинам:
> 
> 1. **Стоимость:** Groq дешевле в 3 раза, чем OpenAI ($0.24 за 1M tokens vs $1 за 1M)
> 2. **Скорость:** Groq быстрее (0.7 сек vs 1.5-2 сек для OpenAI)
> 3. **Модель:** Mixtral-8x7b — это modern MoE (Mixture of Experts), очень хороша для задач с контекстом
> 
> Важно: это не критично, система работает с любым LLM API.
> Я просто выбрал оптимальный вариант для текущих требований."

### Вопрос 4: "Сравните с Terraform + CloudFormation?"

**Правильный ответ:**

> "Terraform и CloudFormation — это tools для развертывания инфраструктуры (IaC).
> AWS Optimizer — это tool для АУДИТА существующей инфраструктуры.
> 
> Они решают разные проблемы:
> - Terraform: 'Как описать инфру в коде?'
> - AWS Optimizer: 'Найди проблемы в моей СУЩЕСТВУЮЩЕЙ инфре'
> 
> На самом деле, они дополняют друг друга."

### Вопрос 5: "Почему TypeScript везде?"

**Правильный ответ:**

> "TypeScript выбран для всего стека (frontend + backend) по пяти причинам:
> 
> 1. **Типизация:** Исключает целый класс runtime ошибок
> 2. **IDE поддержка:** Лучше автозавершение и рефакторинг
> 3. **Конвенция:** Весь фронтенд и бэкенд понимают одинаковые типы
> 4. **Масштабируемость:** При росте проекта с 5k до 50k строк кода TS спасает
> 5. **Индустрия:** TypeScript стал стандартом в production приложениях
> 
> Это не просто 'best practice' — это наука о снижении багов."

### Вопрос 6: "Можно ли использовать это в production?"

**Правильный ответ:**

> "Да, но с предостережениями:
> 
> **Что уже production-ready:**
> - Аутентификация и управление сессиями
> - AWS credentials шифруются AES-256
> - Error handling и retry logic
> - Basic rate limiting на API
> 
> **Что нужно добавить перед production:**
> - Кластеризация MongoDB для high availability
> - Load balancing на backend
> - Redis кеширование результатов сканирования
> - More comprehensive logging и monitoring
> - Более строгие ограничения на API rates
> 
> Это не 'игрушка' — это реальное приложение, требующее typical production upgrades."
