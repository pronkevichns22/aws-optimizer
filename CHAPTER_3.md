## 3.1 Архитектура системы и обоснование выбора технологического стека

### 3.1.1 Структура многоуровневой архитектуры

Разработанный программный комплекс AWS Optimizer построен на основе классической трёхуровневой архитектуры (Model-View-Controller), обеспечивающей чёткое разделение ответственности и независимое масштабирование каждого компонента. Архитектура системы представлена на Рисунке 3.1.

Система состоит из следующих уровней:

**Уровень представления (клиентская часть).** Веб-приложение, разработанное на базе фреймворка React.js версии 19 с использованием языка TypeScript. Приложение функционирует на локальном хосте порт 5173 при разработке. Клиентская часть обеспечивает визуализацию результатов сканирования инфраструктуры, представление метрик в виде интерактивных графиков, реализованных с использованием библиотеки Chart.js, и взаимодействие пользователя с системой.

**Уровень бизнес-логики (серверная часть).** REST API-сервис, реализованный на базе платформы Node.js с использованием фреймворка Express.js и языка TypeScript. Сервис развёртывается на локальном хосте порт 5000 при разработке. На данном уровне реализованы аутентификация и авторизация пользователей, инициирование процессов сканирования инфраструктуры облачного провайдера, применение встроенных правил анализа безопасности и финансовой эффективности, взаимодействие с внешними сервисами (AWS SDK, LLM-модель Groq).

**Уровень хранения данных (база данных).** Документоориентированная база данных MongoDB, функционирующая на локальном хосте порт 27017 при разработке и на управляемом сервисе MongoDB Atlas в production-среде. База данных хранит учётные данные пользователей, историю сканирований, обнаруженные уязвимости, метрики безопасности и финансовой эффективности.

Взаимодействие между уровнями реализуется через стандартные протоколы:
- Frontend ↔ Backend: HTTPS, JSON
- Backend ↔ Database: MongoDB Driver (TCP/IP)
- Backend ↔ External Services: HTTPS (AWS SDK, Groq API)

## 3.2 Программная реализация основных модулей

### 3.2.1 Модуль аутентификации и авторизации

Система безопасности доступа реализована на основе токен-ориентированного подхода с использованием JWT (JSON Web Tokens), подписанных алгоритмом HS256 (HMAC-SHA256). Архитектура аутентификации предусматривает полное отделение приватного ключа подписания на серверной части, что исключает возможность подделки токенов на клиентской стороне.

**Процесс регистрации пользователя:**

При регистрации пользователь предоставляет адрес электронной почты и пароль. Введённый пароль подвергается криптографическому хешированию с использованием алгоритма bcrypt с параметром раунда итераций (cost factor), установленным на значение 10. Это соответствует времени обработки приблизительно 100 миллисекунд, обеспечивая баланс между вычислительной стоимостью (затруднение для攻击者) и приемлемым временем отклика системы. В базу данных сохраняется исключительно хеш пароля; оригинальный пароль не сохраняется и не может быть восстановлен даже в случае компрометации базы данных. Использование автоматически генерируемых солей (salts), встроенных в алгоритм bcrypt, предотвращает атаки на основе предварительно вычисленных таблиц хешей (rainbow tables).

**Процесс аутентификации (вход в систему):**

При входе пользователь повторно предоставляет электронную почту и пароль. На серверной части введённый пароль хешируется тем же алгоритмом bcrypt и сравнивается с сохранённым в базе данных хешем. Если совпадение подтверждено, генерируется JWT-токен с следующими параметрами:

- **Алгоритм подписи:** HS256 (HMAC-SHA256) с использованием серверного секрета
- **Время жизни токена:** 24 часа (требует пересоздания при истечении)
- **Содержимое payload:** userId, email, iat (время создания), exp (время истечения)
- **Хранение на клиенте:** localStorage браузера (защищено Same-Origin Policy)

**Листинг 3.1 — Процесс валидации JWT-токена в API**

```typescript
// middleware/authMiddleware.ts
const verifyToken = (req: Request, res: Response, next: NextFunction) => {
  const authHeader = req.headers.authorization;
  
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing or invalid token format' });
  }
  
  const token = authHeader.substring(7);
  
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET, {
      algorithms: ['HS256'],
      complete: false
    });
    req.user = decoded;
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'Token expired' });
    } else if (err.name === 'JsonWebTokenError') {
      return res.status(401).json({ error: 'Invalid token signature' });
    }
    return res.status(401).json({ error: 'Token validation failed' });
  }
};

export default verifyToken;
```

**Авторизация при обращении к защищённым API:**

Все защищённые эндпоинты (к исключением публичных эндпоинтов регистрации и входа) требуют предоставления действительного JWT-токена в заголовке HTTP Authorization в формате "Bearer <token>". Middleware-функция verifyToken выполняет следующие проверки последовательно:

1. Проверка наличия заголовка Authorization
2. Проверка формата (начиная с "Bearer ")
3. Извлечение токена
4. Верификация подписи (исключает подделку)
5. Проверка срока действия (exp claim)
6. Проверка целостности payload

Если любая из проверок не пройдена, запрос отклоняется с кодом ошибки HTTP 401 (Unauthorized). Успешная валидация добавляет декодированный payload в объект req.user, доступный в последующих обработчиках маршрутов.

### 3.2.2 Модуль сканирования инфраструктуры AWS

Модуль реализует автоматизированный процесс полного аудита облачной инфраструктуры AWS, выявляя как проблемы безопасности, так и неоптимальные расходы. Сканирование является асинхронной операцией, инициируемой пользователем через интерфейс, и не требует наличия постоянного соединения с backend.

**Этап 1: Предварительная подготовка и валидация учётных данных**

При инициировании сканирования пользователь предоставляет AWS credentials (Access Key ID и Secret Access Key), необходимые для аутентификации в AWS API. Credentials передаются по защищённому каналу HTTPS и никогда не сохраняются в файловой системе или логах. На серверной части credentials немедленно шифруются алгоритмом AES-256 (Advanced Encryption Standard, блочный размер 256 бит) с использованием мастер-ключа, хранящегося в переменной окружения. Это гарантирует, что даже при компрометации базы данных монгоБД учётные данные AWS остаются защищёнными. Перед сохранением система проверяет валидность credentials путём выполнения пробного запроса DescribeInstances к AWS API. Если запрос отклонен, пользователю выдаётся понятная ошибка (например, "Credentials are invalid" или "Access denied to EC2"). Это позволяет оперативно выявить проблемы на этапе ввода данных.

**Этап 2: Параллельный поиск ресурсов**

После валидации credentials backend инициализирует AWS SDK v3 с поддержкой параллельных асинхронных запросов. Система одновременно (используя Promise.all в Node.js) отправляет запросы к следующим AWS API операциям:

- **DescribeInstances:** поиск всех EC2 инстансов, включая тип машины, состояние (running/stopped), объём CPU, RAM, сетевые интерфейсы
- **DescribeVolumes:** поиск EBS томов, включая размер (ГБ), тип (gp2/gp3/io1), зашифрован ли, снимки состояния
- **DescribeAddresses:** поиск Elastic IP адресов и проверка, привязаны ли они к инстансам
- **DescribeSecurityGroups:** получение правил входящего/исходящего трафика
- **GetAccountSummary:** статистика IAM ролей, пользователей, политик

Параллелизм критически важен для производительности, так как каждый запрос требует 0.5-2 сек в зависимости от размера ответа и задержки сети. При последовательных запросах время выполнения составило бы 5-10 сек; параллельный подход сокращает его до 2-3 сек.

**Этап 3: Обогащение данных и применение правил**

После получения всех ресурсов система выполняет локальный анализ, применяя встроенный набор правил проверки безопасности и оптимизации затрат. Для каждого обнаруженного нарушения создаётся объект Alert со следующей структурой:

```typescript
interface Alert {
  id: string;                    // Уникальный ID alert
  resourceId: string;            // EC2 instance-id, volume-id и т.д.
  severity: 'CRITICAL' | 'HIGH' | 'MEDIUM' | 'LOW';
  category: 'SECURITY' | 'COST';
  ruleId: number;               // Ссылка на правило (1-20+)
  description: string;           // Понятный текст проблемы
  recommendation: string;        // Действие для исправления
  estimatedSavings?: number;    // USD/месяц если есть финансовое влияние
  detectedAt: Date;             // Время обнаружения
}
```

**Этап 4: Расчёт агрегированных метрик**

После классификации всех alerts система вычисляет:- **Total Infrastructure Spend:** сумма всех потребляемых EC2, EBS, Data Transfer, RDS и т.д. (в USD/месяц)
- **Wasted Spend:** сумма расходов на обнаруженные проблемы (stopped instances, unattached EIP, old snapshots)
- **Health Score:** числовая оценка от 0 до 100, где каждое CRITICAL alert -20 пунктов, HIGH -10, MEDIUM -5, LOW -1
- **Security Score:** отдельная оценка, основанная только на SECURITY alerts
- **Cost Optimization Score:** оценка финансовой эффективности

**Временная характеристика:** Общее время сканирования составляет 3-85 секунд в зависимости от количества ресурсов:
- 50-100 ресурсов: 3-5 сек
- 500-1000 ресурсов: 10-20 сек
- 5000-10000 ресурсов: 40-85 сек

**Листинг 3.2 — Инициирование параллельного сканирования AWS ресурсов**

```typescript
// services/awsScanner.ts - фрагмент
async function scanAWSInfrastructure(region: string, credentials: AWSCredentials) {
  const ec2 = new EC2Client({ region, credentials });
  
  const [instances, volumes, addresses, sgs] = await Promise.all([
    ec2.send(new DescribeInstancesCommand({})),
    ec2.send(new DescribeVolumesCommand({})),
    ec2.send(new DescribeAddressesCommand({})),
    ec2.send(new DescribeSecurityGroupsCommand({}))
  ]);
  
  return { instances, volumes, addresses, sgs };
}
```

**Этап 5: Сохранение результатов**

Все результаты сканирования сохраняются в коллекцию MongoDB Audit с уникальным scanId, временной меткой timestamp и полным списком alerts. Это позволяет пользователю просматривать историю сканирований, видеть тренды улучшения Health Score и отслеживать, какие проблемы были исправлены.

**Листинг 3.2 — Инициирование параллельного сканирования AWS ресурсов**

```typescript
// services/awsScanner.ts
async function scanAWSInfrastructure(region: string, credentials: AWSCredentials) {
  const ec2Client = new EC2Client({ region, credentials });
  
  // Параллельные запросы к различным сервисам
  const [instances, volumes, addresses, securityGroups] = await Promise.all([
    ec2Client.send(new DescribeInstancesCommand({})),
    ec2Client.send(new DescribeVolumesCommand({})),
    ec2Client.send(new DescribeAddressesCommand({})),
    ec2Client.send(new DescribeSecurityGroupsCommand({}))
  ]);
  
  return {
    ec2: instances.Reservations?.flatMap(r => r.Instances || []),
    ebs: volumes.Volumes,
    eip: addresses.Addresses,
    sg: securityGroups.SecurityGroups
  };
}
```

### 3.2.4 Модуль интеллектуального помощника (AI Advisor)

**Назначение и архитектура:**

Модуль AI Advisor предоставляет пользователю персонализированные рекомендации по оптимизации инфраструктуры AWS с использованием больших языковых моделей (LLM). В отличие от классических систем рекомендаций, основанных на жёстких правилах, AI Advisor способен понять контекст инфраструктуры пользователя и предложить творческие, специфичные для бизнеса решения.

**Архитектура взаимодействия (4 этапа):**

**Этап 1: Формирование контекста запроса**

При каждом обращении к AI Advisor система выполняет следующие операции:

1. Извлекает 5-10 наиболее критичных alerts из последнего сканирования
2. Получает историю сканирований пользователя за последние 30 дней (для анализа тренда улучшения)
3. Если пользователь предоставил дополнительный вопрос, добавляет его в контекст
4. Форматирует все данные в единый текстовый промпт (JSON → plain text)

**Этап 2: Отправка на Groq API**

Подготовленный контекст отправляется на облачный API Groq, который обслуживает LLM модель mixtral-8x7b (8 экспертов, 7 млрд параметров). Запрос структурирован как conversation с двумя сообщениями:

- **System message:** инструкция моделью ("You are an AWS security and cost optimization expert. Provide 3-5 concrete, actionable recommendations.")
- **User message:** контекст alerts + вопрос пользователя

Критичные параметры вызова:
- **Model:** mixtral-8x7b-32768
- **Temperature:** 0.3 (низкое значение → детерминированные, фактические ответы вместо творческих)
- **max_tokens:** 512 (ограничение длины ответа для быстроты)
- **top_p:** 0.9 (nucleus sampling для разнообразия, но в пределах логики)

**Этап 3: Получение и парсинг ответа**

Ответ от Groq приходит в JSON формате с массивом choices (обычно 1 вариант). Система извлекает текст из `choices[0].message.content` и проверяет качество:

- Длина ответа не менее 50 символов (исключает пустые/ошибочные ответы)
- Проверка на наличие HTML/JavaScript код (безопасность)
- Парсинг структурированных рекомендаций (если применимо)

**Этап 4: Сохранение и трансляция**

### 3.3.4 Процесс автоматизированного развёртывания (CI/CD)

Для обеспечения надёжности и воспроизводимости развёртываний реализован полнофункциональный CI/CD pipeline на базе GitHub Actions. Каждый push в main ветку инициирует следующую последовательность операций:

**Этап 1: Валидация кода (3-5 минут)**

1. Checkout код из GitHub репозитория
2. Запуск ESLint для статического анализа TypeScript (исключение стиль-ошибок)
3. Запуск unit-тестов (Jest) для Frontend (120 тестов) и Backend (180 тестов)
4. Запуск integration-тестов (проверка API ↔ БД)
5. Запуск E2E-тестов (автоматизированный браузер, сценарии типа "login → scan → check dashboard")
6. Генерация отчётов о покрытии кода (coverage report)

Если любой из этапов не пройден, pipeline останавливается и разработчик получает уведомление (не происходит развёртывания broken code).

**Этап 2: Сборка и публикация (2 минуты)**

1. Сборка Frontend: `npm run build` → статические файлы в dist/ (optimized, minified)
2. Сборка Docker-образа Backend: `docker build -t aws-optimizer-backend:v1.2.3`
3. Сборка Docker-образа Frontend: `docker build -t aws-optimizer-frontend:v1.2.3`
4. Push образов в AWS ECR (private Container Registry)

**Этап 3: Развёртывание в production (2-5 минут)**

1. Kubernetes читает новые версии образов из ECR
2. Инициирует rolling update: старые pods постепенно заменяются новыми
3. Во время обновления 50% pods работают на старой версии, 50% на новой (zero-downtime deployment)
4. Новые pods проходят health checks (проверка /health endpoint)
5. Если pod не отвечает за 30 сек, откатывается к предыдущей версии
6. Metrics отправляются в Prometheus для мониторинга

**Общее время entwicklung-to-production:** ~5-10 минут

### 3.3.5 Мониторинг и logging в production

**Prometheus & Grafana (метрики реального времени):**

Промеthей собирает метрики с интервалом 15 сек:
- HTTP request latency (ms) - среднее, p95, p99
- Database query latency (ms)
- JWT token generation rate
- AWS SDK call success/failure rate
- Error counts по типам (4xx, 5xx)

Grafana отображает эти метрики в виде графиков, позволяя быстро выявить деградацию производительности или всплески ошибок.

**ELK Stack (логирование и аналитика):**

- **Elasticsearch:** индексирует все логи серверной части (~500 KB/день на 100 пользователях)
- **Logstash:** парсит логи, извлекает поля (userId, requestId, error_type)
- **Kibana:** поиск по логам за 2 недели с фильтрацией по severity (DEBUG, INFO, WARNING, ERROR)

Примеры запросов:
- Все ошибки за последний час: `level:ERROR AND @timestamp:>[now-1h]`
- Сканирования дольше 30 сек: `operation:SCAN AND duration_ms:>30000`
- Все обращения конкретного пользователя: `userId:"12345"`

**Alerting (оповещения):**

- Если p95 latency > 2 сек → отправить alert в Slack
- Если error rate > 1% → create PagerDuty incident
- Если disk space < 20% → email to ops team

### 3.2.5 Модуль API endpoints и интеграции

### Интерпретация результатов сканирования

**Severity уровни:**
- **CRITICAL:** Требует немедленного исправления (уязвимость может быть использована для атаки или приводит к критическим потерям)
- **HIGH:** Высокий приоритет (значительный риск безопасности или затраты)
- **MEDIUM:** Средний приоритет (умеренный риск, рекомендуется исправить в ближайшее время)
- **LOW:** Низкий приоритет (небольшое влияние, можно включить в план оптимизации)

**Финансовое влияние:**
- Столбец "Savings" указывает примерную месячную экономию при исправлении проблемы
- "—" означает, что проблема безопасности (финансового прямого влияния нет, но риск высок)
- Значения варьируются в зависимости от размера ресурса и региона AWS

### Процесс классификации alerts

При сканировании инфраструктуры каждое обнаруженное правило присваивает alert объект с информацией о:
1. **Какой ресурс затронут** (EC2 instance ID, volume ID, bucket name и т.д.)
2. **Какое правило нарушено** (с ссылкой на этот реестр)
3. **Какова серьёзность** (CRITICAL/HIGH/MEDIUM/LOW)
4. **Какая категория** (SECURITY или COST)
5. **Какова рекомендация по исправлению** (с шагами)
6. **Сколько можно сэкономить** (если применимо)

Система автоматически рассчитывает **Health Score** (0-100) на основе всех alerts:
- CRITICAL alert = -20 пунктов
- HIGH alert = -10 пунктов
- MEDIUM alert = -5 пунктов
- LOW alert = -1 пункт

Максимальный Health Score 100 означает, что инфраструктура не содержит известных проблем согласно встроенному набору правил.
